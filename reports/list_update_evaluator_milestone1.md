## Policy Interface

Candidate modules expose:

```python
def choose_update(state, request, history):
    ...
```

- `state`: immutable tuple containing the current list order before access.
- `request`: requested item label.
- `history`: immutable tuple of previous requests in the same trace, excluding the current request.
- return value: integer 0-based target index for the requested item after access.
- if the requested item is currently at rank `r`, the target must be in `0..r`.

Invalid return types, out-of-range targets, import failures, or exceptions from `choose_update` mark the policy invalid with a clear error. External policies run in an isolated worker process with a wall-clock timeout and are evaluated twice to reject obvious nondeterminism. The evaluator passes only immutable tuples to candidates and does not expose mutable evaluator state.

## Cost Model

The harness uses the existing `FULL_COST_MODEL_NAME = full_cost_standard_free_after_access`.

- Internal positions are 0-indexed.
- Accessing an item at 0-based rank `r` costs `r + 1`.
- After access, the requested item may move for free to any earlier position, including staying in place.
- Candidate policies cannot perform paid exchanges.
- The offline oracle comparator may use unit-cost adjacent paid exchanges, canonicalized before access by the existing DP.

Tiny hand checks are covered in both the existing tests and the new milestone tests.

## Workload Suites

Available deterministic suites:

- `smoke`: hand-checkable traces for quick CLI/report validation.
- `all_traces_small`: exhaustive nonempty traces for small `n,T`.
- `random_fixed_seed`: pseudo-random traces generated with fixed seed `20260517`.
- `locality_biased`: repeat/neighbor-biased traces generated with fixed seed `314159`.
- `repeated_item`: repeated hot-item traces.
- `phase_shift`: deterministic hot-item phase changes.
- `adversarial`: traces reused from `scripts.list_update.adversarial_traces.worst_traces_by_policy`.
- `milestone1`: combined suite containing the above families.

## Baselines

Implemented under the same `choose_update` interface:

- `move_to_front`
- `transpose`
- `frequency_count`
- `static_safe`

Also included as a trace-aware comparator:

- `static_optimal`: best fixed permutation for each trace. This is not an online policy, and the JSON labels it as a comparator.

## Oracle Integration

Succeeded for the exact offline trace oracle and adversarial trace source:

- per-trace offline optimum is computed through `scripts.list_update.exact_evaluator.offline_optimum`;
- policy metrics include oracle total regret and competitive ratio;
- adversarial workload traces are sourced from the existing hardened finite trace enumerator.

Limit: no public interface was found for a true finite minimax value over online strategy spaces. The JSON report records `finite_minimax_value_available: false` and includes the missing-interface TODO. This does not block baseline-relative scoring.

## Limitations

- External policy isolation is process-level and time-bounded, but it is not a formal sandbox against malicious Python.
- Determinism checking reruns the policy twice; this catches obvious nondeterminism, not all hidden stateful behavior.
- Exact oracle use is limited to the existing small-`n` scaffold.
- `static_optimal` is a trace-aware fixed-order comparator, not a valid online candidate.
- The evaluator is deliberately not optimized for large workloads yet.
